Python设计模式

什么是设计模式

软件工程中，设计模式是指软件设计问题的推荐解决方案。设计模式一般是描述如何组织代码和使用最佳实践来解决常见的设计问题。类似于算法，都是对于某类问题的经验方案。

需要谨记在心的一点是：设计模式是高层次的方案，并不关注具体的实现细节，比如算法和数据结构。对于正在尝试解决的问题，到底何种算法和数据结构最优，则是由软件工程师自己把握。

[ ]为什么说单例模式已经过时了呢，待后续搞明白！

设计模式不是一成不变的，也不是永远的银弹。随着编程语言和观念的演进，一些设计模式也随之过时，甚至成了反模式；另一些则被内置在编程语言中（如迭代器模式）。另外，也有一些新的模式诞生（比如Borg/Monostate）。

关于设计模式的两个误解

  1. 一开始就就应该使用设计模式。经常能看到的一个现象是：开发人员在尝试一下使用自己的方案解决问题之前，就在纠结应该在代码中使用哪种设计模式。这不仅是错误的，而且本身就违背了设计模式的本质。设计模式是在已有的方案之上发现更好的方案，而不是全新发明。（算法也是如此。）一个方案都没有，怎么比较，又何谈找一个更好的呢。先行动起来,用自己的技能尽可能漂亮地解决问题。如果方案足够漂亮灵活，那也就意味着没有必要浪费时间纠结于此。不要为了使用设计模式而使用设计模式，不要为了强迫自己使用设计模式而限制了自己的创造力。
  2. 随处使用设计模式。这会导致方案很复杂！！！接口和分层的冗余往往会带来很高的成本，往往一个更简单直接的方案就足够了。设计模式并不是万能的，仅当代码确实存在坏味道、难以扩展维护时，才有使用的必要。毕竟节约开发成本才是设计模式的初衷。所以要看需求的具体情况来决定是否有使用设计模式的必要。多思考思考你不会需要它。You Are Not Gonna Need It.)和 保持简单直白 KISS原则 。随处使用和过早优化都是误入歧途。

学习计划

我计划首先按照设计模式的类型来分类学习，然后按照《设计模式》那张图的路径研习各个模式之间的关系。

1. 创建型模式：处理对象创建

工厂模式：使用工厂方法和抽象工厂来初始化对象，与直接实例化对象相比有何优势。
建造者模式：对于由多个相关对象构成的对象，如何简化其创建过程。
原型模式：如何克隆一个对象，以省去创建过程。

2. 结构型模式：处理不同实体间的关系

适配器模式：如何以最小的改变实现系统与外来接口的兼容。
修饰器模式：如何不使用继承来增强对象的功能。
外观模式：如何创建单个入口以隐藏系统的复杂性。
享元模式：如何通过复用一个对象池中的对象来提高内存利用率及应用性能。
MVC模式：如何避免业务逻辑与用户界面代码的耦合，提高可维护性。
代理模式：如何增加额外的保护层，提高应用的安全。

3. 行为型模式：处理实体之间的通信

责任链模式：如何向多个权力不同的接收者发送请求。
命令模式：如何让应用能够取消已经执行的操作。
解释器模式：如何基于Python创建一种简单的语言，便于领域专家使用，而无需学习Python编程。
观察者模式：如何在对象发生变化时，通知已注册的相关者。
状态模式：如何创建一个状态机以对问题进行建模，有何优势。
策略模式：如何动态选择处理算法。
模板模式：如何明确区分一个算法的通用与不通用部分，以避免不必要的代码复制。
